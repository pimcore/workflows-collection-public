# AGENTS.md

This file provides guidance to Coding Agents when working with code in this repository.

## Project Overview

Pimcore Documentation Generator — a Docusaurus 3.9.2 multi-version documentation site that aggregates docs from 50+ Pimcore GitHub repositories across multiple platform versions.

## Build & Development Commands

```bash
# Install dependencies (requires Node >= 18.18)
npm install

# Clone repos and prepare versioned docs (must run before build/start)
npm run checkout-doc-versions

# Local development with hot reload
npm start

# Production build (outputs to /build/)
npm run build

# Serve production build locally
npm run serve

# Clear Docusaurus cache
npm run clear
```

## Architecture

### Two-Phase Build Pipeline

1. **`npm run checkout-doc-versions`** — Clones Pimcore repos at specified branches/tags, extracts `/doc` or `/docs` folders, copies them into `versioned_docs/version-X/` directories, and updates `versions.json`.
2. **`npm run build`** — Docusaurus builds the static site from the prepared docs.

### Key Configuration Files

- **`versionConfiguration.js`** — Master config mapping platform versions to repository branches, target directories, sidebar groups, and enterprise flags. This is the primary file to edit when adding/updating versions or repos.
- **`docusaurus.config.js`** — Docusaurus site config (navbar, footer, Algolia search, plugins, edit URL mapping).
- **`sidebar-modificator.js`** — Transforms auto-generated sidebar items into grouped categories based on `sidebarGroup` from versionConfiguration.
- **`sidebars.js`** — Root sidebar config (uses autogeneration).

### Build Scripts (`/bin/`)

- `checkoutDocVersions.mjs` — Main build script orchestrating repo clones and doc extraction.
- `modules/versionConfigHelper.mjs` — Utilities for reading version configuration.
- `modules/git.mjs` — Git clone/checkout operations (supports OAuth token auth via `PERSONAL_ACCESS_TOKEN`).
- `resources/` — Static sidebar templates, index files, and pre-built versioned docs for older versions.

### Custom UI (`/src/`)

- `components/HeroBanner/` — Homepage hero (TSX).
- `components/IndexCards/` — Doc section landing page cards (TSX).
- `clientModules/enterpriseBadge.js` — Client-side module that adds enterprise/professional ribbon overlay to pages.
- `css/custom.css` — Pimcore branding (primary color: `#6428b4`).

### Generated Directories (do not edit manually)

- `/docs/` — Current/next version docs (generated by checkout script).
- `/versioned_docs/` — Historical version docs (generated by checkout script).
- `/versioned_sidebars/` — Per-version sidebar configs (generated).
- `/repos/` — Cloned source repositories (generated).

## Version Configuration Pattern

In `versionConfiguration.js`, each version entry contains:
- `isVersion: true/false` — `true` = released, `false` = "next" (development).
- `repos` — Map of repos with `branch`, `targetDirectory`, `sidebarGroup`, `enterprise` flag, and optional `copyReadme`.
- Target directories use numeric prefixes for ordering (e.g., `01_Pimcore`, `10_Datahub`).

Enterprise flags: `false` = community, `true` = enterprise-only, `'professional'` = professional edition.

## Documentation Refactoring Workflow

For restructuring or improving docs directly in the source repos:

```bash
# 1. Create feature branches across all repos for the latest version
npm run prepare-doc-branches <branch-name>

# 2. Make changes directly in repos/<repo-name>/docs/ (or /doc)

# 3. Build docs from current repo state (no git checkout, uses working tree as-is)
npm run build-docs-from-repos

# 4. Preview locally
npm start
```

- `prepare-doc-branches` clones missing repos, creates/switches to the named branch, and removes stale repo directories not in the latest version config.
- `build-docs-from-repos` extracts docs from `./repos/` into `./docs/` for the latest (next) version only, preserving existing `versioned_docs/` and `versions.json`.
- `dev-docs` runs the initial doc extraction, starts the Docusaurus dev server, and watches `repos/*/docs/` for changes — automatically syncing them into `./docs/` for instant hot-reload.

## Documentation Writing Rules

### Edit in source repos, never in /docs/

The `./docs/` directory is generated — **never edit files there directly**. All documentation changes must be made in the original source repository under `./repos/<repo-name>/docs/` (or `/doc`). The mapping between `./docs/` subfolders and their source repos is defined in `versionConfiguration.js` via the `targetDirectory` property. For example, `./docs/10_Datahub/` maps to the repo `pimcore/data-hub` because its config has `targetDirectory: '10_Datahub'`.

To find which repo owns a doc file: look up the first-level subfolder name in `./docs/` (e.g. `01_Pimcore`) in `versionConfiguration.js` → `repos` → find the entry where `targetDirectory` matches.

### Cross-repository links

Links between docs from different repos (e.g. platform-version to pimcore core) must use **GitHub file URLs** as a convention. The build system automatically rewrites these to correct relative paths during doc extraction.

Write cross-repo links as:

```markdown
[Pimcore Core Framework](https://github.com/pimcore/pimcore/blob/2026.x/doc/README.md)
[Studio Backend API](https://github.com/pimcore/studio-backend-bundle/blob/1.x/doc/README.md)
```

The URL format is: `https://github.com/pimcore/<repo>/blob/<branch>/doc/<path>`

**How it works:** During `prepareVersionDocs` (and `buildDocsFromRepos` / `dev-docs`), a preprocessor detects GitHub URLs matching `github.com/pimcore/<repo>/blob/<branch>/doc/<path>`, looks up the repo in `versionConfiguration.js`, maps the source path to the final `docs/` target directory, and replaces the URL with a correct relative path. The branch in the URL is ignored during mapping - only the repo name and doc path matter.

**Benefits:**
- Links work on GitHub when browsing source repos directly
- Authors don't need to know target directory names or numeric prefixes
- Target directory renames only require updating `versionConfiguration.js`, not every source file
- Docusaurus validates the resulting relative links during build

**Do not** use relative paths like `../../Pimcore/` for cross-repo links (these break because source and target directory names differ). **Do not** use `docs.pimcore.com` absolute URLs (Docusaurus cannot validate these).

Within the same repo, relative links work fine. Always verify with `npm run build`.

### Verify docs build cleanly

After making changes, run `npm run dev-docs` (or `npm run build-docs-from-repos && npm run build`) and check the output for warnings. Docusaurus is configured to warn on broken links (`onBrokenLinks: 'warn'` and `onBrokenMarkdownLinks: 'warn'` in `docusaurus.config.js`). Fix all broken links, dead references, and broken images before considering the work done.

### Line length

Wrap markdown source lines at approximately 120 characters. Break at natural language boundaries (end of sentence, clause, or list item) rather than mid-word. Exceptions where lines may exceed 120 characters:
- Markdown table rows
- URLs and markdown links
- Code blocks (fenced with triple backticks)
- Front matter and HTML tags

### Writing style

Write for developers, solution architects, and decision-makers with a solid technical understanding:
- Use plain English - clear, precise, and to the point
- Keep paragraphs short; prefer bullet lists and code examples over long prose
- Lead with the most important information; omit filler and unnecessary context
- Use active voice and direct instructions ("Run the command", not "The command can be run")
- **Do not use em dashes (`—`)** - use regular dashes (`-`), commas, or restructure the sentence. Em dashes are a telltale sign of LLM-generated text.
- Avoid marketing language. Write technically, not promotionally.
- Use "Pimcore Studio" (not "Studio UI", "admin UI", or "backend") as the consistent term for the administration interface. Exception: page titles may use "Pimcore Studio UI" where the "UI" suffix aids discoverability.

**Style anti-patterns to fix on every pass.** When reworking existing documentation, do not carry over weak prose just because it was already there. Actively rewrite these patterns:

| Anti-pattern | Example | Fix |
|---|---|---|
| Passive "can be" / "is used to" | "Tags can be assigned to elements" | "Assign tags to elements" |
| "You can find X in Y" | "You can find the settings in Pimcore Studio" | "Open the settings in Pimcore Studio" |
| Filler intros | "The following example shows how to..." | Just show the example with a one-line lead-in |
| Vague qualifiers | "really", "very", "some", "powerful", "reasonable", "a lot of" | Remove or replace with specifics |
| Informal hedging | "just", "simply", "take a look into", "let's" | Use direct instructions |
| Circular definitions | "Properties define which properties..." | Rephrase to avoid repetition |
| Wordy "in order to" / "provides the ability to" | "...which provides the ability to create tasks" | "...supports these tasks:" |
| Unnecessary "Note:" prefixes | Starting paragraphs with "Note:" or "Please note:" | Use Docusaurus `:::note` admonitions or integrate into prose |

This applies to every documentation task - writing new content and reworking existing content equally. Do not treat existing prose as acceptable by default; apply the same standard as for new writing.

**Tighten wording, but preserve meaning.** Style improvements must not silently drop content. When you condense a sentence, verify that every distinct piece of information it conveyed survives in the rewrite. "Wordy" and "redundant" are not the same thing - a sentence can be wordy yet carry two separate ideas that both need to remain. If in doubt, keep the information and tighten the phrasing around it rather than cutting it.

### Docs sitemap structure

The file `.resources/structure.md` defines the target structure and sitemap for the documentation. Use it as a guide when creating or reorganizing content. This document is a living reference and will be updated regularly as the docs evolve.

**Rendering:** The site is built with [Docusaurus](https://docusaurus.io/docs). Refer to the Docusaurus documentation and its subpages for details on supported Markdown features, admonitions, tabs, code blocks, front matter, and other content capabilities.

**Directory-based sitemap with numeric prefixes:** The sitemap is defined entirely by the directory and file structure. Use numeric prefixes to control page ordering, and **leave gaps between numbers** so new pages can be inserted later without renaming existing folders/files or updating all cross-links. For example:

```
03_Installation/
05_Configuration/
10_Advanced_Topics/
```

This way, adding `07_Migrations/` later does not require renaming `05_Configuration/` or `10_Advanced_Topics/`.

**`README.md` as index page:** A `README.md` file inside a folder always represents the index (landing) page for that sitemap level. Every folder that should be navigable as a section must contain a `README.md`.

**Structure patterns:**

- **Top-level chapters** (e.g. "Setup", "Platform Version Releases", "Release Notes") come from the `pimcore/platform-version` repository. This repo uses `targetDirectories` in `versionConfiguration.js` to map each source subdirectory to its own root-level doc entry (e.g. `01_Setup.md` → `0001_Setup.md`, `02_Platform_Version_Releases` → `0003_Platform_Version_Releases`). To add or reorganize top-level chapters, edit docs in `repos/platform-version/doc/`.
- **`targetDirectories`** is an alternative to `targetDirectory` — it maps multiple source subdirectories/files from a single repo to separate target directories, each with optional `sidebarGroup`. Repos using the existing `targetDirectory` string work unchanged.
- **All other repositories** are nested as: `sidebarGroup` → repository name (derived from `targetDirectory`). For example, a repo with `sidebarGroup: 'Data Onboarding & Distribution'` and `targetDirectory: '0201_Datahub'` appears as "Datahub" under the "Data Onboarding & Distribution" group.

When creating new content, consult `.resources/structure.md` and determine whether it belongs in a top-level chapter (`pimcore/platform-version`) or under a specific repo's section based on this hierarchy.

**Backlog:** The `repos/platform-version/doc/backlog/` directory contains source content that has been drafted but not yet placed into the documentation structure. When working on a chapter, check the backlog for relevant content that should be incorporated or adapted.

**Content placement for cross-cutting concerns:** Some topics span multiple chapters (e.g. Platform Version touches both Installation and Updating). Define one canonical location for the detailed content and link from other locations. Do not duplicate procedural content across chapters.

### Review with a context-free sub agent

After writing or significantly editing documentation, spawn a **context-free sub agent** (using the Agent tool) to review the result. The sub agent must not share your conversation context — it receives only the doc files to review. Its job is to critically evaluate:

- **Readability** — Is the text clear, well-structured, and easy to follow for a developer new to the topic?
- **Through line** — Does the documentation flow logically from start to finish? Are there gaps, abrupt transitions, or sections that feel disconnected?
- **Completeness** — Are there unexplained concepts, missing steps, or dead-end references?

The sub agent must provide concrete, actionable feedback. Apply its suggestions, then re-submit to the sub agent for another round. **Loop until the sub agent is satisfied with the result, up to a maximum of 5 rounds** - do not skip this step or accept the first draft.

When spawning the review sub agent, always provide:
1. The file paths to review
2. The documentation goal (one sentence, e.g. "guide a developer through a complete skeleton installation")
3. The writing standards summary (from this file)
4. Cross-references the reviewer should check (e.g. "verify links to files X and Y exist on disk")
5. Whether the content was adapted from existing source material (so the reviewer can flag outdated references like old UI names or removed features)

### Pimcore product knowledge

When writing about Pimcore, keep these facts accurate:

- **Open core, not open source.** All Pimcore source code is released under the Pimcore Open Core License (POCL). A substantial set of components is openly available; enterprise editions provide access to additional modules. Do NOT say "GPLv3" or "open source".
- **PXM (Product Experience Management)** is the overarching category for Pimcore. It should be prominent in overview and introductory content.
- **Pimcore is not "not an out-of-the-box product."** Pimcore provides solid out-of-the-box and configurable functionality (no coding needed) plus full developer extensibility. Do not frame it as developer-only or requiring code to get started.
- **Assets: embedded metadata is read-only.** Pimcore reads and displays embedded metadata (EXIF, IPTC, XMP) but cannot edit it. Custom metadata fields are separate and fully editable.
- **Pimcore Studio URL** is `/pimcore-studio` (not `/admin`). The old `/admin` route was for the legacy Admin Classic UI.
- **Technology foundation** (PHP, Symfony, MySQL, etc.) belongs in architecture documentation, not in overview pages.
- **Graphics:** prefer reusing existing images from the academy (`.resources/`) or existing docs. Mermaid diagrams are supported by both Docusaurus and GitHub. Use `<div class="image-as-lightbox"></div>` before a Markdown image to render it as a clickable lightbox.

### Restructuring existing documentation

When restructuring or rewriting an existing page (as opposed to writing new content), follow these additional rules:

- **No information loss.** Every piece of technical content in the original must appear in the rewrite - links, parameter descriptions, edge-case notes, cross-references. Tightening wording is fine; dropping information is not. After rewriting, generate a `git diff` of the file and review it section by section to confirm nothing was silently removed.
- **Verify code examples against source code.** When a page contains code examples that reference Pimcore APIs (Twig helpers, PHP interfaces, service definitions), verify function signatures, parameter names, and return types against the actual source in `repos/` before finalizing. Do not assume the original documentation was correct.
- **Provide the original to the review sub-agent.** When spawning the review sub-agent for restructured content, also provide the `git diff` or the original file content so the reviewer can flag any dropped information, not just evaluate the new version in isolation.

### Bundle documentation structure

Bundle docs follow the same shape. Apply it when restructuring a bundle:

- **Installation is a folder, Upgrade is its subpage.** `doc/00_Installation/README.md` plus
  `doc/00_Installation/01_Upgrade.md`. Never leave `Upgrade` as a top-level chapter.
- **Extension points live in the feature's Customization chapter**, not in a separate top-level page. A feature with a
  single extension point still gets `05_Customization.md` with the topic as its first subchapter.
- **Permissions are documented once, in the Installation chapter.** Do not repeat the table anywhere else.
- **There is no "Studio Integration" chapter.** See below.

### Do not document the internal Studio API

Nearly every bundle had a hand-maintained page dumping its Pimcore Studio REST endpoints. This is an anti-pattern:

- The controllers are marked `@internal` and change without deprecation. We explicitly tell readers not to call them,
  so a request/response reference has no audience.
- Studio Backend already serves generated OpenAPI docs at `/pimcore-studio/api/docs`. A hand-written mirror only rots.
- The endpoint base paths, permission tables and "internal, do not use" warnings were duplicated across two or three
  pages of the same bundle.

Delete these pages. Keep only the genuinely actionable part, the **pre-response events**, and move them into the
Customization (or Technical Aspects) chapter with a short `:::warning` that the endpoints themselves are internal.

### Documenting events

Event lists in existing docs are routinely incomplete. Do not copy them forward. Instead:

- Enumerate `src/Event/**` and document **every** event, grouped by what it does.
- Verify each one is actually **dispatched**. A class can exist and never be fired (a `postCsvItemExport` event was
  documented for years but never dispatched). Grep for the dispatch site, and watch for **import aliases**, which make a
  naive grep report a live event as dead.
- Take the payload/schema from the event class **constructor**, not from the old docs.
- Skip events on `@internal` classes and internal plumbing (for example interceptor-proxy save hooks). They are not
  extension points.

### Screenshots

Screenshots are content, and are held to the same accuracy bar as prose.

- **Never use throwaway demo data.** A config named `Docs_Screenshot_Demo` or `car-file-export-demo` must never appear
  in a published image. Use the demo's real, meaningful configurations.
- **The configuration in the shot must actually be configured.** An inactive config with no description, no columns and
  no workspace illustrates nothing, and silently contradicts the text describing those settings. Fill it in, save it,
  then capture.
- **Restore the demo afterwards.** Delete every configuration you created and verify it is gone. Pimcore Studio also
  persists the **panel layout per user**, so reopen any element tree or side panel you closed. Leave the demo as you
  found it.
- **The public demo is shared and churns.** Other people leave throwaway configurations behind (`sdfsdf`, `test`), and
  they appear and vanish between sessions. Check the configuration tree before shooting. Do **not** delete data you did
  not create without asking, someone may be mid-work on it; surface it and let the user decide.
- **Do not fabricate a state you cannot reach.** If a screenshot would need a running job the demo cannot run, remove
  the image and describe the behavior in text. Never ship an image whose caption contradicts its content.
- **Check for duplicates.** Three "different" screenshots turned out to be byte-identical copies of the same empty tab.
- **Do not blindly restore an "original" screenshot.** Older images are frequently Admin Classic (ExtJS), which no
  longer exists in 2026.x. Verify before reverting, and recapture in Pimcore Studio instead.
- **Before replacing a screenshot, recover what the original one showed.** `git log --oneline --all -- <image>`, then
  `git show <prev-commit>:<image> > /tmp/orig.png` and open it. The filename and the alt text are not the intent, and
  neither is whatever a previous recapture left behind. This has gone wrong twice, in the same way both times: an image
  named `filtering.png` with alt text `Filtered Grid` was replaced by a GraphiQL pane showing the query, when the
  original was the object grid with the column filter applied. The screenshots on a query-sample page usually illustrate
  the **data in Pimcore** that the query reads, not the query itself. Replacing them with a picture of the query makes
  every one of them redundant with the code block printed right below it, which is the tell that the intent was lost.
- **An annotated screenshot is carrying an argument.** Purple boxes, arrows and highlighted rows mark exactly the thing
  the prose is pointing at, for example the three grid rows that a `limit 3, offset 1` query returns, or the two buttons
  that open the GraphQL Playground. A clean recapture that drops the annotation silently deletes the explanation.
  Reproduce the annotation, or the image is worse than the one it replaced.
- **A batch recapture that collapses two images into one byte-identical file has failed.** After recapturing, diff the
  set: `md5sum doc/img/**/*.png | sort | uniq -Dw32`. Two distinct pages showing the same screenshot means at least one
  capture never happened.

Framing and size:

- **Capture at 1440x900.** Bigger viewports (1920, let alone 2560) shrink the controls until the reader cannot make them
  out. A larger window is not a better screenshot.
- **Close the element trees** before capturing so the configuration panel gets the width. Click the panel toggle icons in
  the bottom-left rail (the flexlayout border buttons), which is the product's own control. Never hide or remove nodes
  from the DOM.
- **Crop to the region that matters**, and let the height vary per image. A form tab whose fields end halfway down should
  not ship 400px of empty canvas, but a taller form must not be cropped so aggressively that its last fields are lost.
  Check each crop by reading it back.
- **Do not guess crop offsets, measure them.** When a page has several stacked sections and each one has its own image,
  scroll the section heading to the top, then read the bounding boxes of that heading and of the **next** heading, and
  crop between them. Guessing pixels is what makes an image bleed into the next section or start mid-row.
- **Expand the dropdown or menu the text is about.** If the prose enumerates the modes of a select, the screenshot should
  show them.
- **Every control the image is about must actually be set.** An empty select or an unconfigured dialog illustrates
  nothing and contradicts the caption above it. If the shot is of the "Workflows" setting, pick a workflow.
- **Close any menu you opened for a previous shot.** A stray open dropdown left hanging over an unrelated tab was the
  single most common defect across these repos.

Capture technique (Playwright MCP):

- **Disable spellcheck before capturing:** `document.querySelectorAll('textarea, input').forEach(e => e.setAttribute('spellcheck', 'false'))`, then blur the focused field. Otherwise
  prose you typed into a description field renders with red squiggles.
- **Modals:** screenshot the **element** `.ant-modal-content`, not the viewport and not `.ant-modal`. The viewport cuts
  tall modals, and `.ant-modal` includes wrapper padding that leaves a dark strip of canvas at the bottom. If the modal
  is taller than the viewport, enlarge the viewport first (`browser_resize`) so the whole element renders. When the doc
  wants the surrounding context visible behind the dialog, capture the viewport instead and crop.
- **Fonts:** `await page.evaluate(() => document.fonts.ready)` before capturing. A missing webfont renders text in a
  serif fallback and looks like a product styling bug.
- **Toasts** auto-dismiss and animate. Wait for the selector, then ~700ms for the slide-in to settle, then screenshot
  the viewport and crop, rather than element-screenshotting mid-animation.
- **Menus:** hover the entry the text is about, not a neighbouring one, and move the real mouse to it. A hover applied
  and then released does not survive to the screenshot.
- **Panel state:** test flexlayout border buttons with `className.includes('--selected')`. Matching on `'selected'`
  alone also matches `--unselected`, which silently reports every panel as open.
- **The Playwright viewport is independent of the visible browser window.** Setting a viewport larger than the window
  looks broken to a human watching, but captures render at the full viewport size regardless. Say so rather than letting
  it read as a bug.
- **Saving a configuration returns you to the General tab.** Recapture the tab you wanted afterwards, or you will ship a
  General screenshot labelled as something else.
- **Nested tabs:** `.ant-tabs-tabpane-active` matches both the outer config tab and the inner settings tab. Scope to the
  last **visible** pane (`filter(p => p.offsetParent !== null)`) or you will read rows from the wrong tab.
- **Select dropdowns:** options from every select accumulate in the DOM, so query only inside the open dropdown
  (`.ant-select-dropdown:not(.ant-select-dropdown-hidden)`). Long lists are virtualised, so type to filter rather than
  scanning for an option that is not rendered yet.
- **`scrollIntoView` invalidates coordinates.** Scroll first, wait, then measure the bounding box, then click.
- **Mask any live secret** (API keys, webhook URLs) visible on the public demo, and report it so the key can be rotated.
  This has to be checked per image: the same key was masked in one screenshot and left in plain text in its neighbour.

Two traps that cost real time:

- **The orphan check must include the repo root `README.md`.** With `copyReadme: true` it becomes the chapter index, so
  an image referenced only from it looks unreferenced when you grep `doc/**/*.md`. Deleting it breaks the build. This
  caught me twice (`overview.jpg` in data-hub, `webhook-overview.png` in webhooks, which is a designed graphic, not a
  stale screenshot).
- **Deleting on the shared demo:** the configuration tree caches. An entry still showing after a delete is not proof the
  delete failed, reload the page to check. And before clicking any delete, identify the exact button (the config panel
  footer, not a grid row) and confirm the dialog names your own configuration.

### The docs build panics on rspack's persistent cache

`experimental_faster` enables rspack's persistent cache. When files are deleted between builds it panics with
`should mgm exist` at `rspack_core/src/cache/persistent/...`. This is not a content error, it reproduces with your
changes reverted. Fix:

```bash
rm -rf .docusaurus node_modules/.cache build && npm run build
```

Do this from the docs-generator root, not from a repo under `repos/`. A purged build also surfaces broken links that the
cached build silently skipped.

### HTML5 drag-and-drop in Pimcore Studio

Several Studio features (tree to workspace, GraphQL operators onto fields) require an HTML5 drag. Neither synthetic
`PointerEvent`s nor Playwright's `dragTo()` / the Chrome DevTools `drag` tool complete the drop. This sequence works:

```js
const dt = new DataTransfer()
const fire = (type, el, x, y) => el.dispatchEvent(
  new DragEvent(type, { bubbles: true, cancelable: true, composed: true, clientX: x, clientY: y, dataTransfer: dt, view: window })
)
fire('dragstart', sourceEl, sx, sy)          // the app fills `dt` itself
fire('dragenter', window, cx, cy)            // window-level matters
fire('dragover',  window, cx, cy)
const under = document.elementFromPoint(cx, cy)   // resolve the real drop zone
fire('dragenter', under, cx, cy)
fire('dragover',  under, cx, cy)
fire('drop',      under, cx, cy)
fire('dragend',   sourceEl, cx, cy)
```

Three things make it work:

1. dispatching `dragenter`/`dragover` at the **window** level, not only on the target;
2. dropping on `document.elementFromPoint()` rather than the element you assume is the target;
3. **querying the drop zone only after `dragstart`**, with a short `await` in between. The drop zones
   (`[class*="dnd"]`, `.hotspot-droppable`) are rendered *during* the drag, so a reference captured beforehand points at
   nothing. This was the difference between the drop failing and landing.

### Moving a page breaks cross-repo links

Renaming or moving a doc file usually breaks a link from `repos/platform-version` (the release notes link to each
bundle's upgrade page). Always run a full `npm run build` after a restructure and fix every reported broken link,
including in other repos.

### Accuracy and source verification

Do not write documentation based on assumptions. Before documenting behavior, APIs, or configuration:
- Check the actual source code in the relevant repository to verify correctness
- Cross-reference with existing docs, README files, and inline code comments
- If something is unclear or potentially incorrect, raise it in the chat rather than guessing
- Never fabricate code examples, config options, or API signatures - every claim must be backed by a verified source
- All references to Admin Classic UI and ExtJS have to be replaced with corresponding replacements in Pimcore Studio
- Before documenting that a package needs to be installed separately, check the consuming package's `composer.json` to verify it is not already a default dependency. Use `gh api repos/pimcore/<repo>/contents/composer.json` to check remote repos not cloned locally.
- Check the backlog directory (`repos/platform-version/doc/backlog/`) and existing chapters for content that overlaps with what you are writing. Avoid creating duplicate procedural content.
- **Permissions:** do not assume they live in a `PermissionConstants` class. Some bundles register them from a service constant via raw SQL in the installer (direct-edit uses `PimcoreBackendPermissionService::PERMISSION_NAME` and a `REPLACE users_permission_definitions` statement). Grep for `users_permission_definitions`, `Permission\Definition` and `IsGranted` as well, or you will wrongly conclude a bundle has no permissions. Then verify the constant, that the installer actually creates it, and **where it is enforced**. A permission enforced on PHP controllers (`#[IsGranted]`) protects the API; one checked only in the frontend (`isAllowed(...)` in `assets/studio/js/...`) only hides UI. Describe the observable effect, not the constant name. Do not present a constant name as if it were the permission key; they often differ.
- **Install prerequisites:** a Composer dependency is not an installed bundle, and neither is a registered dependent
  bundle. `registerDependentBundles` (`DependentBundleInterface`) only loads a bundle into the kernel, and
  `pimcore:bundle:install` never cascades. Every Datahub adapter (File Export, Simple REST, Webhooks, Data Importer)
  needs `bin/console pimcore:bundle:install PimcoreDataHubBundle` run separately, because Datahub's own installer is
  what creates the `plugin_datahub_config` permission its controllers require. Wording like "Composer pulls it in
  automatically, no manual bundle ordering is required" reads as "it is set up for you" and is a trap. When a bundle
  depends on another, check whose installer creates the state it needs.
- **Verify UI mechanisms in the UI.** A doc claimed a field is renamed with an "Operator Alias". Opening the dialog
  showed there is no such control: an advanced column has a **Title**, **Source fields** and **Transformers**. Source
  then confirmed the header comes from the column label. Do not describe a dialog you have not opened.
- **Read the predicate, not the prose.** Behavioural claims are where the docs are most often wrong, and the bug is
  usually one operator deep:
  - webhooks: "Include unpublished ... auto-saved versions are not" was false. `matchByPublished()` returns
    `$includeUnpublished || ($published && !$versionOnly)`, and `||` short-circuits, so enabling the toggle also lets
    version-only (auto-save) saves through.
  - webhooks workspaces: paths are matched with a bare `str_starts_with`, with no folder-boundary check, so an entry for
    `/Product Data` also matches `/Product Data Archive`. "Everything below it" was the wrong mental model.
  - simple-rest output formatters: the loader is `$formatters[$f->getType()] = $f` in a loop. There is no priority, so
    the last registration for a type silently wins. "Replaces the shipped behaviour" overstated a guarantee the code
    does not make.
  When the docs describe a rule, open the method that implements it and read it line by line.
- **Legacy icons hide in prose.** Old pages inline ExtJS icons as images (`![Add button](../img/Icons/add_button.svg)`,
  a green circle plus; a yellow gear). Studio uses neither. Grep for `img/Icons/` and name the control in words instead.
- **Storage:** do not assume a feature is served by the search index. Alternative Element Trees, for example, build one generated table per configuration (`bundle_bpt_aet_ott_<id>`) and query it through DBAL; only a column filter bridges into the Generic Data Index. Read the listing service before describing where data comes from.

### Reference values (Pimcore 2026.x)

These values are verified and should be used consistently:
- Pimcore Studio URL path: `/pimcore-studio`
- Docker PHP image tag: `pimcore/pimcore:php8.5-latest`
- Minimum PHP version: 8.5
- Skeleton package: `pimcore/skeleton` (includes `pimcore/platform-version` by default)
- Demo enterprise package: `pimcore/demo-enterprise` (includes Studio and all dependencies pre-configured)
- Search engine requirement: OpenSearch >= 2.7 or Elasticsearch >= 8.0.0

## CI/CD

GitHub Actions workflow (`.github/workflows/generate-documentation.yml`) runs on schedule (every 4h weekdays) and on manual dispatch. It checksout docs, builds, uploads to GitHub Releases, and triggers Algolia crawler reindex.
